# Docker 測驗

## Question 1
在生產環境中使用 `latest` 標籤的主要風險是什麼？

- [x] 可能意外拉取到未經測試且可能破壞應用程式的新版本
- 它會導致 Docker 引擎強制將容器轉換為虛擬機器運行
- 使用該標籤會導致容器自動獲得主機的 root 存取權限
- Docker 官方禁止在公共註冊表中使用此標籤

**Hint:** 思考一下當映像檔發行商更新了標籤指向的內容，而你卻執行了自動拉取時會發生什麼事。

## Question 2
Docker 的「多階段建置」 ($Multi-stage$ $builds$) 主要能帶來哪種好處？

- [x] 減少最終映像檔的大小，僅保留執行所需的檔案
- 允許單個容器同時運行多個獨立的作業系統程序
- 自動將容器部署到 Kubernetes 等編排平台
- 透過加密所有映像檔層來提高傳輸過程的安全性

**Hint:** 考慮建置應用程式所需的工具（如編譯器）在程式執行時是否仍然必要。

## Question 3
在 Docker Compose 的健康檢查配置中，`start_period` 參數的用途為何？

- [x] 提供一段寬限期，在此期間內的檢查失敗不會計入重試次數上限
- 設定健康檢查指令執行時允許的最大超時秒數
- 指定兩次健康檢查動作之間的間隔時間
- 定義容器在被標記為不健康之前必須連續失敗的次數

**Hint:** 某些應用程式在正式提供服務前需要執行密集的初始化任務。

## Question 4
關於「使用者定義橋接網路」 ($User-defined$ $bridge$ $networks$)，下列敘述何者正確？

- [x] 提供自動的內建 DNS 解析，允許容器透過名稱互相通訊
- 它會自動禁用容器的所有外部互聯網存取以提高安全性
- 此網路類型僅支援在 Docker Swarm 模式下運行
- 每個使用者定義網路都必須手動設定靜態的 $MAC$ 地址才能運作

**Hint:** 想想為什麼在 Docker Compose 中，你可以直接使用資料庫的服務名稱來建立連線。

## Question 5
在開發環境中，為什麼通常建議對原始碼使用「綁定掛載」 ($Bind$ $mounts$) 而非「具名卷」 ($Named$ $volumes$)？

- [x] 它允許主機上的檔案更改立即反映在容器中，支援即時重載
- 綁定掛載由 Docker 完全管理，因此在跨平台遷移時更具移植性
- 它能自動解決主機與容器之間的檔案讀取權限衝突問題
- 綁定掛載會將資料儲存在主機記憶體中以大幅提升 $I/O$ 效能

**Hint:** 考慮開發過程中需要频繁修改代碼並立即看到結果的需求。

## Question 6
在 Dockerfile 中使用 `USER` 指令的主要目的是什麼？

- [x] 降低安全風險，避免以具有高權限的 root 身份運行程序
- 自動將主機上的所有用戶帳號同步到容器內部的作業系統
- 加速映像檔的建置速度，因為非 root 用戶建置的層較少
- 強制 Docker 守護程序在特定硬體資源限制下運行

**Hint:** 如果一個受損的程序以最高管理權限運行，其對系統造成的潛在威脅會增加。

## Question 7
關於 `COPY` 與 `ADD` 指令的差異，下列敘述何者最準確？

- [x] `COPY` 僅支援基本的本地檔案複製，而 `ADD` 具備自動解壓縮與 URL 下載功能
- `ADD` 指令會自動驗證遠端檔案的數位簽章，而 `COPY` 不會
- `COPY` 只能將檔案複製到映像檔的唯讀層，而 `ADD` 則支援寫入層
- Docker 官方強烈建議棄用 `COPY` 指令並全面改用 `ADD` 以提升效能

**Hint:** 其中一個指令被設計為「簡單且僅用於本地複製」，而另一個則包含更多高級自動化功能。

## Question 8
根據 Kubernetes 概述，下列何者屬於其「自我修復」 ($Self-healing$) 的範疇？

- [x] 重新啟動失敗的容器，並在節點失效時重新排程 Pod
- 自動檢測並修復應用程式原始碼中的邏輯漏洞
- 自動掃描並移除主機上的物理硬體灰塵以防止過熱
- 在主機作業系統崩潰後自動重裝整個 $Linux$ 核心

**Hint:** 關注平台如何確保你設定的副本數量與實際運行的容器數量保持一致。

## Question 9
當執行命令 `docker run -p 8080:80 nginx` 時，具體發生了什麼映射？

- [x] 將 Docker 主機的 $8080$ 連接埠流量轉發到容器內部的 $80$ 連接埠
- 將容器的 $8080$ 連接埠流量導向主機的 $80$ 連接埠
- 將容器的資源使用限制在主機總記憶體的 $80\%$ 以內
- 建立一個具備 $8080$ 個節點的集群來運行單個 $Nginx$ 容器

**Hint:** 連接埠映射的格式通常是 `[主機連接埠]:[容器連接埠]`。

## Question 10
為了讓建置過程更高效，為什麼在 Dockerfile 中應該將 `COPY . .` 指令盡可能往後放？

- [x] 避免因程式碼頻繁變動而導致前面層的建置快取 ($Cache$) 失效
- 確保所有檔案在複製前都已經過 `RUN` 指令的病毒掃描
- 降低映像檔的最終大小，因為最後一層會被 Docker 自動壓縮
- 防止原始碼中的敏感資訊被包含在早期的中間映像檔層中

**Hint:** Docker 在構建映像檔時會嘗試重用之前已經生成的穩定層。
